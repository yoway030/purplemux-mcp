# Stage 1 · Canonical Extraction — purplemux CLI Feature Inventory

**purplemux** `0.3.2` (Node ≥ 20). Binaries: `purplemux`, `pmux` → `bin/purplemux.js`.
This is the **authoritative merged** output of the 3-agent extraction panel
(Sonnet / Opus / Codex-gpt5.5-high), reconciled by the orchestrator and, where the
panel disagreed, resolved **empirically against the live server** (`PMUX_PORT=16500`).

The purplemux CLI is a **thin wrapper over a localhost HTTP API**. Everything below is
what an MCP server must reproduce. **Recommendation: the MCP server should call the HTTP
API directly, not shell out to `purplemux`** — this avoids the CLI's argv-joining and
`stripFlags` quirks and recovers response bodies the CLI discards (e.g. `tab close`).

---

## 1. Auth & connection model

| Item | Value | Source |
|---|---|---|
| Base URL | `http://localhost:${PORT}` (localhost only, plain HTTP) | `cli.js:21` |
| PORT resolution | `PMUX_PORT` env → else `~/.purplemux/port` (trimmed; empty→null) | `cli.js:19` |
| TOKEN resolution | `PMUX_TOKEN` env → else `~/.purplemux/cli-token` | `cli.js:20` |
| Auth header | `X-Pmux-Token: <TOKEN>` on **every** request (guide docs lowercase `x-pmux-token`; case-insensitive) | `cli.js:41,59` |
| JSON requests | also send `Content-Type: application/json` (raw screenshot path omits it) | `cli.js:41` |
| Missing PORT | CLI dies: `PMUX_PORT not set and ~/.purplemux/port missing (is the server running?)` | `cli.js:29` |
| Missing TOKEN | CLI dies: `PMUX_TOKEN not set and ~/.purplemux/cli-token missing (is the server running?)` | `cli.js:30` |
| Bad/missing token (server) | **HTTP 403** `{ "error": "Forbidden" }` on every `/api/cli/*` route | server bundle |
| Token file | auto-generated by server if absent: 32-byte hex, `~/.purplemux/cli-token`, mode `0600`, compared with `timingSafeEqual` | server bundle |
| Wrong HTTP method | **405** `{ "error": "Method not allowed" }` + `Allow` header | server bundle |

`requireEnv()` runs at the start of every networked command; `help` needs neither PORT nor
TOKEN. The CLI reads PORT/TOKEN once at module load.

## 2. Command surface & HTTP mapping

Routing is two-layer: `purplemux.js` decides CLI-vs-server (`CLI_COMMANDS =
{workspaces, tab, memory, mem, api-guide, help}`), then `cli.js main()` dispatches.
Bare `purplemux` / `purplemux start` boots the app server (out of scope).

| # | CLI command | HTTP | Query / Body | Success response |
|---|---|---|---|---|
| 1 | `workspaces` | `GET /api/cli/workspaces` | — | `{ workspaces:[{ id, name, directories[] }] }` |
| 2 | `tab list [-w WS]` | `GET /api/cli/tabs[?workspaceId=WS]` | `workspaceId?` | `{ tabs:[{ tabId, workspaceId, name, sessionName, panelType, agentProviderId, agentSessionId }] }` |
| 3 | `tab create -w WS [-n NAME] [-t TYPE]` | `POST /api/cli/tabs` | body `{ workspaceId, name?, panelType? }` | **201** `{ tabId, workspaceId, paneId, sessionName, name, panelType, agentProviderId, agentSessionId }` |
| — | *(no CLI cmd)* tab info | `GET /api/cli/tabs/<tabId>?workspaceId=WS` | `workspaceId` | `{ tabId, workspaceId, paneId, name, sessionName, panelType, agentProviderId, agentSessionId }` |
| 4 | `tab send -w WS TAB_ID CONTENT…` | `POST /api/cli/tabs/<tabId>/send?workspaceId=WS` | body `{ content }` | `{ status:"sent" }` |
| 5 | `tab status -w WS TAB_ID` | `GET .../status?workspaceId=WS` | `workspaceId` | `{ tabId, workspaceId, alive, command?, cliState?, agentProviderId, agentSessionId, claudeSessionId }` |
| 6 | `tab result -w WS TAB_ID` | `GET .../result?workspaceId=WS` | `workspaceId` | `{ content }` (current pane snapshot) |
| 7 | `tab close -w WS TAB_ID` | `DELETE /api/cli/tabs/<tabId>?workspaceId=WS` | `workspaceId` | `{ ok: boolean }` (CLI prints literal `ok`, **discards body**) |
| 8 | `tab browser url -w WS TAB_ID` | `GET .../browser/url?workspaceId=WS` | `workspaceId` | `{ tabId, url, title }` |
| 9 | `tab browser screenshot -w WS TAB_ID [-o FILE] [--full]` | `GET .../browser/screenshot?workspaceId=WS[&full=1][&format=base64]` | `workspaceId`, `full?`, `format?` | file mode → `{ saved, bytes }`; base64 mode → `{ tabId, format:"png", base64 }`; raw → `image/png` bytes |
| 10 | `tab browser console -w WS TAB_ID [--since MS] [--level LVL]` | `GET .../browser/console?workspaceId=WS[&since][&level]` | `workspaceId`, `since?`, `level?` | `{ tabId, entries:[{ level, text, ts, source?, url?, line? }] }` (≤500) |
| 11 | `tab browser network -w WS TAB_ID [--since][--method][--url][--status]` | `GET .../browser/network?workspaceId=WS[&…]` | filters | `{ tabId, entries:[{ requestId, method, url, status?, mimeType?, resourceType?, error?, ts, endedAt? }] }` (≤500) |
| 12 | `tab browser network -w WS TAB_ID --request RID` | `GET .../browser/network?workspaceId=WS&requestId=RID` | `requestId` | `{ tabId, requestId, body }` (cached after 1st call) |
| 13 | `tab browser eval -w WS TAB_ID EXPR…` | `POST .../browser/eval?workspaceId=WS` | body `{ expression }` | `{ tabId, value }` (CDP `Runtime.evaluate`, `returnByValue`+`awaitPromise`, **10s**) |
| 14 | `api-guide` | `GET /api/cli/api-guide` | — | `text/markdown` (printed raw) |
| 15 | `help` / `-h` / `--help` | *(local, no HTTP)* | — | static usage text |

## 3. Enums, defaults, and the full status-code inventory

**`panelType` enum (6):** `terminal` · `claude-code` · `codex-cli` · `agent-sessions` ·
`web-browser` · `diff`. **Default when omitted = `terminal`.** Invalid → **400**
`{ error:"Invalid panelType", validPanelTypes:[…] }`.

| Status | Body / meaning | When |
|---|---|---|
| 200 / 201 | normal / created-tab JSON | success |
| 200 `image/png` | raw PNG bytes | screenshot without `format=base64` |
| 400 | `workspaceId is required` / `content is required` / `expression is required` | missing body field (server-side defense in depth) |
| 400 | `Invalid panelType` (+ `validPanelTypes`) | `tab create` bad type |
| 400 | `Tab is not a web-browser panel` | any `browser` cmd on a non-browser tab |
| 403 | `Forbidden` | bad/missing token |
| 404 | `Workspace not found` | `tab create` unknown workspace |
| 404 | `Tab not found` | tab lookup fail (unknown `tab list -w` workspace returns `{tabs:[]}`, not 404) |
| 404 | `Response body unavailable` | `network --request` body not cached/available |
| 405 | `Method not allowed` (+ `Allow`) | wrong HTTP verb |
| 409 | `Tab session is not running` | `send`/`result` on a dead tmux session |
| 409 | `Browser tab not attached yet` | browser cmd before webview `dom-ready` |
| 409 | `<capture/eval error message>` | screenshot capture fail / eval JS exception or timeout |
| 409 | `agent-not-installed` / `agent-path-missing` (+ provider metadata, `suggestedCommand`) | `tab create` for claude-code/codex-cli whose CLI binary is missing |
| 500 | `No pane available in workspace` / `Failed to create tab` | create failures |
| 503 | `Browser bridge unavailable (Electron-only feature)` | any `browser` cmd in headless / non-Electron mode |

## 4. Semantics & edge cases the MCP must honor

1. **`tab send` auto-submits — EMPIRICALLY CONFIRMED (live).** Content is delivered as a
   terminal **bracketed paste** (`ESC[200~ … ESC[201~`) and the server then **presses
   Enter** (observed: Enter, ~600ms wait, Enter again). Test: sending `echo <mark>` with
   **no** trailing newline caused the command to *execute* (mark appeared twice: the typed
   line + its output). **⇒ MCP `send` must NOT append `\n` to submit — the server does it.**
   Appending your own newline *may* cause a spurious extra Enter (two reviewers' live retests
   did not cleanly reproduce a double-submit artifact, so this is a caution, not a confirmed
   failure). **Safety rule for Stage 2: `pmux_send_input` should normalize by stripping a
   single trailing `\n` from `content` by default.** (This resolves the panel's only material
   disagreement: Sonnet/Opus inferred "no auto-Enter" from the API guide; Codex read the
   server bundle correctly; the live test — reconfirmed independently by all reviewers —
   proves Codex right.)
1b. **`claudeSessionId` is a legacy alias of `agentSessionId`** in `tab status` (identical
   value in the current server build) — Stage 2 should not model them as two independent
   fields. **Unverified-in-headless:** the enriched screenshot (`{tabId,format:"png",base64}`)
   and eval (`{tabId,value}`) shapes come from the server bundle, not the frozen api-guide
   (`{base64}` / `{tabId,value}`); do not hard-code the extra keys without a live Electron
   check — read defensively.
2. **`web-browser` tabs are Electron webviews, not tmux.** `tab status.alive` is normally
   **`false`** for them and must NOT be read as "dead" — probe health via browser endpoints.
   `tab close` on a browser tab skips the tmux kill. `tab result` is not meaningful for them.
3. **Two browser failure tiers:** `503` = not running under Electron at all (hard, don't
   retry); `409 "not attached yet"` = right runtime but webview not `dom-ready` (transient,
   retry shortly — common right after creating a `web-browser` tab).
4. **Ring buffers = last 500** for both console and network, per tab; `--since MS` is the
   incremental polling cursor (invalid `since`→0 server-side).
5. **Screenshot has 3 shapes:** `-o FILE` (raw PNG to disk → `{saved,bytes}`), default
   base64 JSON (`{tabId,format:"png",base64}`), or direct `image/png` bytes via the API.
   `--full` = full-page beyond viewport (`full=1`; server also accepts `full=true`).
6. **network:** `--method` upper-cased server-side; `--url` substring; `--status` exact int;
   `--request RID` switches to single-body mode and ignores the list filters; body cached.
7. **`memory` / `mem` are DEAD** — listed in `purplemux.js` `CLI_COMMANDS` but have no case
   in `cli.js main()` and **no `/api/cli/memory` route** exists (verified live: `error:
   unknown command: memory`). **Do NOT expose an MCP tool for them.**
8. **CLI quirks (reasons to hit HTTP directly):**
   - `stripFlags` treats `--full` as if it takes a value → a token right after `--full` is
     swallowed. Only safe when `--full` is last.
   - `tab send` / `browser eval` rejoin argv with single spaces → **whitespace/newlines
     collapse**; pass exact strings via JSON to preserve them.
   - `tab close` discards the real `{ok:boolean}` body and always prints `ok` (even on
     `{ok:false}`). MCP should surface the real boolean.
   - Flags are space-separated only (`--flag value`), **no `--flag=value`**.

## 5. Canonical candidate MCP tool set (15)

Direct HTTP client, shared: base `http://localhost:$PMUX_PORT`, header `X-Pmux-Token`,
PORT/TOKEN resolved as §1. `*` = required. Cross-cutting error surfacing for every tool:
403 (auth → point at env/`~/.purplemux`), 404 (not found), 409/503 (typed conditions),
ECONNREFUSED (server not running).

| MCP tool | Maps to | Inputs | Output |
|---|---|---|---|
| `pmux_list_workspaces` | `workspaces` | — | `{ workspaces:[{id,name,directories}] }` |
| `pmux_list_tabs` | `tab list` | `workspaceId?` | `{ tabs:[…] }` (cross-workspace if omitted) |
| `pmux_create_tab` | `tab create` | `workspaceId*`, `name?`, `panelType?`(enum, default terminal) | created-tab obj; surfaces 400 `validPanelTypes`, 409 `suggestedCommand` |
| `pmux_get_tab` | *(API-only)* | `workspaceId*`, `tabId*` | tab-info obj |
| `pmux_send_input` | `tab send` | `workspaceId*`, `tabId*`, `content*` (exact string, **auto-submitted**) | `{ status:"sent" }` |
| `pmux_tab_status` | `tab status` | `workspaceId*`, `tabId*` | status obj (note browser `alive:false`) |
| `pmux_capture_pane` | `tab result` | `workspaceId*`, `tabId*` | `{ content }` |
| `pmux_close_tab` | `tab close` | `workspaceId*`, `tabId*` | `{ ok:boolean }` (real body) |
| `pmux_browser_url` | `tab browser url` | `workspaceId*`, `tabId*` | `{ tabId,url,title }` |
| `pmux_browser_screenshot` | `tab browser screenshot` | `workspaceId*`, `tabId*`, `full?`, `savePath?` | `{base64}` or `{saved,bytes}` |
| `pmux_browser_console` | `tab browser console` | `workspaceId*`, `tabId*`, `since?`, `level?` | `{ entries[] }` (≤500) |
| `pmux_browser_network` | `tab browser network` | `workspaceId*`, `tabId*`, `since?`, `method?`, `url?`, `status?` | `{ entries[] }` (≤500) |
| `pmux_browser_network_body` | `tab browser network --request` | `workspaceId*`, `tabId*`, `requestId*` | `{ requestId, body }` |
| `pmux_browser_eval` | `tab browser eval` | `workspaceId*`, `tabId*`, `expression*` | `{ value }` (CDP 10s) |
| `pmux_api_guide` | `api-guide` | — | markdown text (self-doc / introspection) |

**Excluded:** `help` (static text), `memory`/`mem` (dead), `start` (boots the app; MCP
connects to an already-running server). An optional 16th diagnostic tool
`pmux_connection_info` (baseUrl / port source / token presence — never leak the token) is a
nice-to-have for troubleshooting; decide in Stage 2.
